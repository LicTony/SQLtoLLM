## 1. Objetivo del producto

Desarrollar una aplicación de escritorio en **C# (WPF)** que permita a un usuario extraer contexto estructural y lógico de una base **SQL Server** (solo lectura) y generar un texto optimizado para ser copiado y pegado en un LLM, con el fin de:

- Entender esquemas
- Analizar lógica existente
- Generar scripts (`ALTER` / `INSERT` / DDL)
- Realizar refactors asistidos por LLM

> ⚠️ El LLM **NO** tiene acceso directo a la base de datos.  
> ⚠️ La aplicación **NO** ejecuta scripts de modificación.

---

## 2. Alcance

### ✅ Incluido

- **Motor:** SQL Server (MSSQL) como único motor inicial.
- **Conexión vía:**
  - Connection string completo
  - Server + Database + User + Password
- **Extracción de:**
  - `TABLE` (con columnas, PK, IDENTITY, DEFAULT)
  - `FOREIGN KEYS`
  - `INDEXES`
  - `VIEWS`
  - `STORED PROCEDURES`
  - `TRIGGERS`
- Output en **texto plano estructurado**, listo para copy-paste.
- Reordenamiento lógico del output.

### ❌ Excluido explícitamente

- Ejecución de `ALTER`, `INSERT`, `DELETE`, etc.
- Persistencia de archivos.
- Análisis automático de dependencias.
- Acceso directo del LLM a la base de datos.
- Otros motores (Oracle / PostgreSQL quedan fuera de esta versión).

## 4. Flujo funcional (User Journey)

### 4.1 Selección de conexión

```
Usuario selecciona modo de conexión
    ├── "Connection String"
    └── "Credenciales" (Server + Database + User + Password)
         ↓
    Presiona [Conectar]
         ↓
    ✅ Éxito → Se habilita la sección de objetos
    ❌ Error  → Se muestra mensaje de error claro; no se avanza
```

### 4.2 Ingreso de objetos

El usuario pega una lista de objetos en un campo de texto libre:

```
dbo.Clientes, dbo.SP_CalcularSaldo, dbo.VW_Saldos
```

La aplicación:

1. Hace `split(',')` y limpia espacios.
2. Consulta el catálogo de SQL Server.
3. Detecta si el objeto existe y si hay ambigüedad de tipo.

### 4.3 Grilla de resolución

| Nombre objeto | Tipo detectado | Tipo editable | Estado |
|---------------|---------------|---------------|--------|
| `dbo.Clientes` | TABLE | TABLE | ✅ Resuelto |
| `dbo.Log` | *(vacío)* | VIEW / TABLE | ⚠️ Ambiguo |
| `dbo.XYZ` | — | — | ❌ No existe |

**Reglas:**
- Si algún objeto tiene tipo vacío o estado `❌`, el botón **Ejecutar** se deshabilita.
- El usuario puede corregir manualmente el tipo en la columna "Tipo editable".

### 4.4 Ejecución

Al presionar **[Ejecutar]**, la aplicación:

1. Toma los objetos validados de la grilla.
2. Los reordena según el orden lógico definido (ver sección 5).
3. Ejecuta el query de extracción contra SQL Server.
4. Genera un único bloque de texto estructurado.
5. **Copia el resultado al portapapeles automáticamente.**

---

## 5. Orden lógico del output

El output **no** respeta el orden de entrada del usuario, sino el siguiente orden fijo:

| Prioridad | Tipo de objeto |
|-----------|---------------|
| 1 | `TABLE` (incluye PK + FKs) |
| 2 | `INDEX` |
| 3 | `VIEW` |
| 4 | `TRIGGER` |
| 5 | `PROCEDURE` |

> **Motivo:** este orden mejora el razonamiento del LLM y la coherencia en la generación de scripts, ya que las tablas base siempre preceden a las vistas y procedimientos que las referencian.

---

## 6. Formato del output

El output es texto plano estructurado, diseñado para ser resistente al copy-paste desde cualquier herramienta (SSMS, Notepad, LLM).

```text
================================================================================
OBJECT TYPE : TABLE
OBJECT NAME : dbo.Clientes
================================================================================
CONTEXT:
CREATE TABLE dbo.Clientes (
    ClienteID   INT          NOT NULL IDENTITY(1,1),
    Nombre      VARCHAR(100) NOT NULL,
    ...
    CONSTRAINT PK_Clientes PRIMARY KEY (ClienteID)
);

ALTER TABLE dbo.Clientes
ADD CONSTRAINT FK_Clientes_Pais FOREIGN KEY (PaisID) REFERENCES dbo.Paises(PaisID);
================================================================================


================================================================================
OBJECT TYPE : PROCEDURE
OBJECT NAME : dbo.SP_CalcularSaldo
================================================================================
CONTEXT:
CREATE PROCEDURE dbo.SP_CalcularSaldo
    @ClienteID INT
AS
BEGIN
    ...
END;
================================================================================
```

**Características del formato:**

- ✅ Texto plano puro
- ✅ Sin XML ni JSON
- ✅ Sin caracteres especiales que rompan el copy-paste
- ✅ Separadores visuales claros para el LLM

## 7. Requisitos no funcionales

| Requisito | Detalle |
|-----------|---------|
| Compatibilidad SQL Server | 2016 SP3 o superior |
| Permisos requeridos | Solo `SELECT` + `VIEW DEFINITION` |
| Funciones SQL prohibidas | Sin `STRING_AGG` (no disponible en 2016) |
| Collation | Manejo explícito de `COLLATE DATABASE_DEFAULT` |
| Dependencias externas | Ninguna |
| Performance | Tiempo de ejecución aceptable para decenas de objetos |
| Seguridad | Sin escritura en base de datos |

```
┌─────────────────────────────────────────────────────────┐
│                      UI (WPF)                           │
│  • Ventana única con secciones progresivas              │
│  • Grilla editable de resolución de objetos             │
└────────────────────┬────────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────────┐
│                      Core                               │
│  • Modelo común de objetos                              │
│  • Orden lógico del output                              │
│  • Formateo de salida (text plain)                      │
└────────────────────┬────────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────────┐
│               Provider MSSQL                            │
│  • Detección de tipo de objeto (catálogo SQL Server)    │
│  • Ejecución del query de extracción                    │

```

> **Extensibilidad:** El diseño de providers separados permite agregar en versiones futuras soporte para **Oracle** y **PostgreSQL** sin modificar el Core ni la UI.

## 10. Riesgos y mitigaciones

| Riesgo | Probabilidad | Mitigación |
|--------|-------------|------------|
| Output demasiado grande para el LLM | Media | El usuario decide qué objetos incluir; no hay extracción masiva automática |
| SPs muy largas que excedan el contexto del LLM | Media | Uso consciente por parte del usuario; futura feature: truncado opcional |
| Errores de collation en objetos con nombres especiales | Baja | Forzado a `COLLATE DATABASE_DEFAULT` en todas las comparaciones |
| Uso indebido en ambientes PROD | Baja | Solo lectura por diseño; sin posibilidad de ejecución de DDL/DML |

---

## 11. Criterios de éxito

| Criterio | Verificación |
|----------|-------------|
| El usuario puede copiar el output y pegarlo en un LLM | Test manual con ChatGPT / Claude / Copilot |
| El LLM entiende el contexto sin pedir aclaraciones | Evaluación subjetiva con casos reales |
| El LLM puede generar `ALTER` / `INSERT` / DDL coherentes | Test con al menos 3 casos de uso reales |
| No se ejecuta ningún cambio automáticamente | Verificación de permisos y código (solo `SELECT`) |
| La conexión falla con mensaje claro si las credenciales son incorrectas | Test de credenciales inválidas |

---

## 12. Conclusión

Este producto **no automatiza decisiones**, no ejecuta cambios, y no viola controles de seguridad.

Es una herramienta de **extracción de contexto**, diseñada para potenciar el razonamiento humano asistido por LLM.

| Dimensión | Evaluación |
|-----------|-----------|
| Arquitectura | ✅ Correcto — separación clara UI / Core / Provider |
| Operación | ✅ Seguro — solo lectura, sin persistencia |
| Técnico | ✅ Sólido — restricciones de SQL Server 2016 contempladas |
| Documentación | ⚠️ Pendiente — Query de extracción debe incorporarse al PRD |
